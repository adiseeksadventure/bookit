# BOOKIT - FULL STACK TAKE HOME ASSIGNMENT

## Overview

Build a full-stack web application called "BookIt", a live event booking platform similar to a simplified Eventbrite.

Time Limit: 3 Days

Required Stack:
- React or Next.js
- Node.js
- Express.js
- PostgreSQL

The application should allow organizers to create events with limited seating and users to browse and book those events.

---

# User Roles

There are two roles:

## User
- Sign up
- Log in
- Browse events
- Search events
- Book seats
- View bookings
- Cancel bookings

## Organizer

An organizer can do everything a normal user can, plus:

- Create events
- Edit events
- Set event capacity
- View attendees
- View analytics for their events

Authentication can use a single user table with a role field:
- user
- organizer

---

# Required Features

## 1. Authentication

Implement:

- Sign up
- Log in
- Log out

Requirements:

- Email/password authentication
- Passwords must be hashed using bcrypt or argon2
- Never store plaintext passwords
- Protected routes
    - Guests cannot create events
    - Guests cannot make bookings

---

## 2. Browse and Search Events

Create an events listing page showing:

- Title
- Date
- Time
- Venue
- Seats Remaining
- Price

Support:

- Search by title
- Filter by date
- Pagination

Pagination example:
20 events per page

Important:

The system should scale to over 100,000 events.

Do NOT:

- Load everything into memory
- Filter in JavaScript

Instead:

- Use indexed SQL queries
- Efficient pagination

Sold-out events must:

- Be clearly marked
- Not allow bookings

---

## 3. Event Detail and Booking

Each event page should display:

- Full description
- Venue
- Date
- Capacity
- Live seats remaining

Booking requirements:

Booking creates:

- One confirmed booking

Critical Business Rule:

If capacity = N

Confirmed bookings must NEVER exceed N.

Concurrency requirement:

If two users try booking the last seat simultaneously:

Exactly ONE booking succeeds.

The other request must receive a clean "Sold Out" response.

The reviewers will test concurrent requests.

Additional rule:

A user cannot book the same event twice.

---

## 4. My Bookings

Users can view all bookings.

Display:

- Event details
- Booking status

Allow:

- Cancel booking

Cancellation must:

- Delete/cancel booking
- Return the seat back to available inventory

---

## 5. Organizer Dashboard

Organizer can:

Create Event

Fields:

- Title
- Description
- Venue
- Date/Time
- Capacity
- Price

Edit Event

Constraint:

Capacity can NEVER become smaller than the number of already confirmed bookings.

Organizer Dashboard should show:

- Organizer's events
- Bookings sold per event

Organizer can also:

- View attendee list

---

## 6. Analytics

For every organizer event show:

- Total Views
- Bookings Started
- Confirmed Bookings
- View → Booking Conversion Rate

Analytics must be calculated from an Activity Log table.

---

# Database Requirements

Use PostgreSQL only.

Minimum tables:

users
- Accounts
- Role

events
- Organizer owner
- Capacity
- Event information

bookings
- User
- Event
- Booking relationship

Must enforce:
No overselling.

activity_log

Append-only table.

Events include:

- event_viewed
- booking_started
- booking_confirmed
- booking_cancelled

Analytics must be computed from this table.

# Suggested API

Authentication

POST /auth/signup

POST /auth/login

POST /auth/logout

Events

GET /events
Supports:

- search
- date
- page

GET /events/:id

Should log:

event_viewed

Booking

POST /events/:id/book

This is the concurrency-critical endpoint.

Bookings

DELETE /bookings/:id

Should:

- Cancel booking
- Free seat

User

GET /me/bookings

Organizer

POST /organizer/events

PATCH /organizer/events/:id

GET /organizer/events

Returns organizer events plus sold counts.

GET /organizer/events/:id/attendees

GET /organizer/events/:id/analytics

Returns analytics computed from activity_log.

Use proper HTTP status codes.

Examples:

401 Unauthorized

409 Conflict (Sold Out)

Return meaningful JSON error responses.
